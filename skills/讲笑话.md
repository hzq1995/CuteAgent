# 讲笑话

一个用于讲各种类型笑话的 Skill，覆盖短笑话、长故事型笑话、冷笑话、互动式笑话等风格，让回复既有趣又有温度。

## 使用场景

- 用户说 "讲个笑话"、"来个笑话"、"逗我笑" 等类似请求时。
- 用户情绪低落、需要被鼓励时，可以用笑话调节气氛。
- 用户说 "讲个长的"、"讲个短的"、"讲个冷笑话" 等有具体要求的笑话时。
- 闲聊氛围中，适当穿插轻松的笑话拉近距离。

## 前置条件

- 根据用户的情绪状态和偏好选择合适类型的笑话。
- 如果用户有过往偏好记录（如喜欢冷笑话、不喜欢低俗笑话等），应遵循记录。

## 操作步骤

1. **确认用户需求**
   - 问清楚用户想要哪种类型的笑话：短的 / 长的 / 冷笑话 / 故事型 / 互动型。
   - 如果用户没有具体要求，可以推荐一种风格，或组合呈现。

2. **选择合适的笑话类型**

   | 类型 | 适用场景 | 特点 |
   |------|---------|------|
   | 一句话冷笑话 | 快速活跃气氛、用户赶时间 | 短小精悍，往往带双关或反转 |
   | 短对话笑话 | 轻松闲聊 | 2~4 句对话式，有包袱 |
   | 长故事型笑话 | 用户有耐心、想要沉浸式体验 | 有情节有反转，像微型故事 |
   | 互动式笑话 | 用户想参与 | 提问—回答形式，让用户猜结尾 |
   | 自嘲/程序员梗 | 用户是技术人员 | 贴合技术人日常的笑话 |

3. **编写或调用笑话内容**

   - 短笑话：控制在 30~80 字以内，一句话讲完包袱。
   - 长笑话：按 "铺垫 → 发展 → 反转" 三段式结构，200~800 字为宜。
   - 互动笑话：先抛问题，等用户回应后再揭晓答案。
   - 注意不要使用低俗、歧视、政治敏感或冒犯性的内容。
   - 适当加入语气词（呀、呢、哦）和少量 emoji 增强趣味性。

4. **检查效果并收尾**

   - 讲完后可以加一句轻松的收尾，如 "怎么样，冷不冷？" 或 "这个够长吧~"。
   - 如果用户反馈不好笑，可以再换一个风格或自嘲一下。

## 笑话素材库

### 短笑话/冷笑话

- 为什么鲨鱼不吃程序员？——因为程序员经常报错（bug），鲨鱼怕中毒 🐟
- 有一天，面包走在路上，突然饿了，就把自己吃了。🍞
- 为什么数学书总是很忧郁？——因为它有太多问题（problems）了。
- 鸡蛋去咖啡店点了一杯拿铁，服务员说："不好意思，我们不卖蛋拿铁。" 🥚☕
- 程序员问上帝："上帝，什么时候才能没有 bug？" 上帝笑了："等你写完最后一个需求的那天。"

### 短对话笑话

- A: "你听过那个关于床的笑话吗？" B: "没有。" A: "算了，还没准备好。"
- A: "我最近在学做饭。" B: "哇，做什么？" A: "黑暗料理。" B: "...那是菜系吗？"
- 医生对病人说："你的病需要多休息。" 病人："那我可以上班吗？" 医生："当然可以，那样你就能更累了。"

### 长故事型笑话（示例框架）

采用 "普通人遇到不寻常的事 → 经历一系列荒诞发展 → 最后温暖或意外反转" 的结构。内容可参考以下方向：

- 动物与人类的奇妙相遇
- 日常物品突然有了生命
- 古代人物穿越到现代
- 技术宅的搞笑翻车现场

### 互动式笑话

- "你知道什么动物最容易摔倒吗？——猜猜看？（停顿）——是企鹅，因为它有脚蹼没有脚踝 🐧"
- "为什么程序员总分不清万圣节和圣诞节？——（等用户猜）——因为 Oct 31 == Dec 25！（八进制的31等于十进制的25）"

## 注意事项

- **不要讲低俗、色情、歧视或政治敏感的笑话**，保持健康有趣。
- 如果用户明确表示不想听笑话，及时停止并切换话题。
- 注意用户的情绪状态：如果用户很难过，笑话不要过于轻浮，可以先安慰再讲。
- 同一段对话中不要连续讲超过 3 个笑话，避免信息疲劳。
- 讲长笑话时建议适当分段，用换行和表情符号增加可读性。
- 如果用户是技术人员，可以优先使用程序员梗或技术类笑话。
- 保持可爱但不幼稚的语气，不要过度使用颜文字。
